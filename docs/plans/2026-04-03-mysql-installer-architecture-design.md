# MySQL Installer 设计决策文档

日期：2026-04-03

## 需求摘要

用户希望在现有 MySQL 离线安装包基础上，补全以下能力：

1. 通过外层命令开关控制监控、日志、备份、压测等能力。
2. 默认这些能力都开启。
3. 帮助文档、架构解析、注释都改为中文。
4. 备份同时支持 NFS 和 S3。
5. 充分考虑缺少 CRD、后补能力、重装复用 PVC 等细分场景。

## 关键决策

### 决策 1：`install` 作为重复执行的对齐动作

原因：

1. 可以把“首次安装”和“后续补装能力”统一到同一条路径。
2. 便于在后续集群能力变化时进行补齐。
3. 降低运维理解成本。

影响：

1. 需要在 `install` 中主动清理已关闭能力的旧资源。
2. 文档必须强调 `install` 的幂等语义。

### 决策 2：实例级能力优先 sidecar，集群级能力不强装

适用对象：

1. mysqld-exporter
2. Fluent Bit

原因：

1. sidecar 与 MySQL 生命周期一致。
2. 对离线环境更友好。
3. 不需要本安装器去安装集群公共能力。

影响：

1. ServiceMonitor 只在 CRD 存在时创建。
2. 后续如需 Prometheus Operator / 集中日志平台，建议独立安装。

### 决策 3：备份后端抽象为 nfs / s3

原因：

1. NFS 与对象存储在客户现场都很常见。
2. 统一命令入口更易理解。
3. 恢复逻辑可共享大部分脚本。

实现方式：

1. NFS 直接挂载卷读写。
2. S3 通过 `minio/mc` 容器完成同步。
3. S3 备份前预取远端快照，再执行本地保留策略。

### 决策 4：默认保留 PVC

原因：

1. 大部分“重装”诉求本质上是“重建控制面资源，保留数据”。
2. 删除 PVC 是高风险动作，应显式触发。

影响：

1. `uninstall --delete-pvc` 才删除数据卷。
2. 文档必须明确说明 PVC 复用条件。

## 方案边界

当前版本不做：

1. 自动安装 Prometheus Operator。
2. 自动安装集群级日志平台。
3. 自定义备份工具镜像构建。

当前版本已覆盖：

1. NFS / S3 备份后端。
2. 中文帮助和中文文档。
3. 安装、恢复、压测、PVC 复用策略。
